# 인사이트 카드 생성기

리서치 쇼가 아니라 의사결정을 돕는 카드를 만든다. 독자는 요청자의 동료와 임원이다.

## 입력

- 회사명 + 주제(아래 9개 중 1개 이상). 주제 무지정 시 어떤 주제가 필요한지 되묻는다. **주제가 여러 개면 주제당 카드 1장**을 따로 만든다 (카드 1장 = 회사 × 주제 × 날짜).
- 요청자가 내부 자료를 붙여넣으면 그것을 1순위 근거로 쓴다 (출처 표기: "내부 자료", type: internal).

## 주제 9개와 슬러그

기업 개요(overview) / 산업·시장(industry) / 재무상태(financials) / 신사업·투자(growth) / 리스크(risk) / 경쟁사 비교(competitors) / 상품(product) / 고객(customer) / 임상·과학적 근거(clinical)

**슬러그는 파일명에만 쓴다.** frontmatter의 `topic`에는 한국어 문자열을 정확히 쓴다 (예: `topic: 기업 개요`).

## 리서치 규칙

1. 모든 중요한 주장에는 출처를 붙인다.
2. 최신 데이터 우선, 오래된 데이터는 시점 명시.
3. 반대 근거와 리스크를 반드시 포함한다.
4. 사실 / 추론 / 추천을 구분해서 쓴다.
5. 요약이 아니라 "그래서 어떻게 봐야 하는가"까지 쓴다.

## 주제별 확인 소스

- 재무상태·기업 개요 → 한국 공시 대상 기업: DART (dart.fss.or.kr) / 해외: 회사 IR·뉴스룸, PR 와이어, SEC 공시
  - sec.gov 원문 링크는 자동 접근이 차단(403)되는 경우가 많다 — 회사 IR 페이지나 PR Newswire의 같은 발표문을 대신 쓴다
- 산업·시장·신사업·리스크 → 뉴스 검색 (최근 12개월)
- 상품 → 공식 사이트·판매 페이지
- 고객 → 유튜브·인스타·리뷰 (VOC, 실사용 후기)
- 임상·과학적 근거 → 논문·임상 등록·규제기관 인증 현황 (한국: 식약처). 규제·임상 데이터가 없는 산업(유통·서비스 등)이면 이 주제 대신 다른 주제를 제안한다
- 경쟁사 비교 → 위 소스를 경쟁사에도 동일 적용
- 대상 회사의 산업에 맞지 않는 소스는 같은 성격의 소스로 치환한다 (예: 유통사의 "임상" → 소비자 리포트)

## URL 검증 (절대 규칙)

카드에 쓰는 모든 출처 URL은 **WebFetch로 실제 접속해서 내용 확인**한 것만 기록한다.

- 접속 실패/내용 불일치 → 그 출처는 버린다
- 필수 소스(DART 등)가 검증에 실패하면 → 같은 사실을 담은 검증 가능한 대체 출처(뉴스, 회사 IR)로 커버한다. 대체 출처도 없으면 그 주장은 카드에서 뺀다
- 검색 결과에 나온 URL이라도 접속 확인 전에는 쓰지 않는다
- verified 판정은 카드에 기재한 출처 기준: 전부 접속 확인 → `verified: true`, 하나라도 미확인 → `verified: false`

## 출력

카드 규격은 **이 스킬 폴더 안의 SCHEMA.md**를 읽고 정확히 따른다 (frontmatter 필수 필드·YAML 주의사항 포함).

- 위치: 프로젝트 루트 `cards/{companySlug}-{topicSlug}-{YYYYMMDD}.md`. `cards/` 디렉토리가 없으면 만들고, 스킬 폴더의 SCHEMA.md를 `cards/SCHEMA.md`로 복사해 둔다 (다음 카드부터 규격 참조용)
- companySlug: 회사 공식 영문 표기(도메인·브랜드명) 기준 소문자-하이픈 (예: 올리브영 → oliveyoung). 기존 카드가 있으면 그 표기를 따르고, 첫 카드면 채택한 슬러그를 요청자에게 한 줄 보고한다
- 같은 회사×주제 기존 카드는 덮어쓰지 않는다 — 다른 날짜의 새 파일이 히스토리가 된다. 단, 오늘 날짜 파일이 이미 있으면 그 파일을 갱신한다
- 숫자 추이가 있으면 chart, 숫자 지표가 있으면 keyMetrics(최대 4개, 증감 방향 포함), 고객·상품 주제에 검증된 평점/리뷰 데이터가 있으면 reputation을 채운다
- 고객 주제면 본문에 `## 고객의 소리` 섹션
- 각 source에 type을 반드시 붙인다 (dart/news/youtube/instagram/official/internal)
- companySlug 결정이 애매한 회사(병기 표기가 여럿)면 공식 도메인의 표기를 우선하고, 채택한 슬러그를 요청자에게 보고한다. company 필드에는 요청자가 부른 이름을 쓴다

## 로고 (선택)

프로젝트에 로고 디렉토리가 이미 있으면 (예: `site/public/logos/`, `public/logos/`) 회사 공식 사이트나 파비콘에서 받아 `{companySlug}.png`로 저장한다. 로고 디렉토리가 없으면 이 단계를 통째로 건너뛴다 — 새로 만들지 않는다.

## 품질 게이트 (제출 전 자가 점검)

- [ ] 모든 숫자에 출처 또는 "추정" 표시가 있는가
- [ ] 반대 근거·리스크가 1개 이상 있는가
- [ ] 모든 출처 URL을 WebFetch로 접속 확인했는가
- [ ] frontmatter가 SCHEMA.md 규격과 일치하는가 (`topic`은 한국어 문자열 9개 중 하나 정확히)
- [ ] 한 줄 요약만 읽어도 결론이 전달되는가
